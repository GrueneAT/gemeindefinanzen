01Strategy

Read this first — the rest is detail.

Strategy summary

Konvergenz statt Refactor. Die lokale --web-*-Schicht ist hex-identisch zur upstream --gat-web-*-Schicht (Researcher: 26 von 26 Tokens drift-frei), und die lokalen Komponenten-Regeln waren 1:1 die Vorlage, aus der DS v2 .gat-panel/.gat-metric-card/.gat-callout/.gat-section-head/.gat-hero gebaut hat. Migration ist deshalb in 95 % der Faelle Such-Ersetz, nicht Design-Entscheidung.

Reihenfolge ist durch CONTEXT.md Decision 1 fixiert: 5 Phasen, je ein atomarer Commit, jeweils mit gruener Test-Suite. Davor (Task 0) ein Baseline-Snapshot-Commit, danach (Task 6) Snapshot-Diff + Audit. Sieben Tasks gesamt, klare lineare Abhaengigkeit.

  • Eine Welle, nicht gestaffelt.

    Die User-Entscheidung im ISSUE ist explizit "Volle Migration in einer Welle" — kein Doppel-Schicht-Halten, keine schrittweise Konvergenz ueber mehrere Issues hinweg.

  • Tab-/Switcher-Block in dashboard.css muss entfernt werden, nicht nur ueberschrieben.

    Doppel-Klassen .tab-btn gat-tab bringen optisch nichts, solange die spaeter geladene Vendor-Regel .tab-btn { ... } die gleiche Spezifitaet hat und gewinnt. DS-Default ist 1:1 identisch — lokaler Block geht ersatzlos raus.

  • INK-Shape-Mismatch in den Chart-Modulen.

    App nutzt semantische Rollen (green/blue/orange/ red/soft/paper), DS nutzt tonale Klassen (text/soft/mute/hairline/ gridline/axis/green/clay/slate). Loesung: Import-Alias INK as DS_INK, lokaler App-Adapter mappt PALETTE[*] auf App-Rollen. Naive Direkt-Uebernahme { INK } bricht die Charts (Serien werden grau).

  • HC-Toggle FOWT-Prevention.

    <head>-Inline-Skript setzt die Klasse auf <html> (nicht <body>, das existiert noch nicht) vor dem ersten Paint, mit try/catch wegen localStorage-Sperre im Privatmodus. DS-Selektor .gat-mode-hc body greift trotzdem (Ancestor-Match).

  • Snapshots nur Baseline + Final, nicht zwischen den Phasen.

    CONTEXT-Decision 2: zu viel Pflege fuer den Marginalwert. Playwright/Chromium, 1440x900, Fixture VA-2026-Auflage.pdf, 8 Bilder (Landing + 7 Tabs). Pixel-Diff <5 % in DS-bewussten Stellen erlaubt.

03Decisions

Key decisions & options

Migrations-Reihenfolge: 5 Phasen, je ein atomarer Commit

Chosen (CONTEXT)

Tokens -> Komponenten -> Charts -> A11y -> Doku. Jede Phase liefert einen gruenen Testlauf und einen vyz9q:-Commit. Baseline-Snapshot davor, Final-Snapshot danach.

Alles in einem Commit — unsauber zu reviewen, wenig Rollback-Granularitaet.

Iterative Doppel-Schicht (App + DS parallel pflegen) — widerspricht dem expliziten Konvergenz-Ziel des Issues.

Wert-Drift zwischen --web-* und --gat-web-*

Chosen (Research)

Researcher hat alle 26 Token-Hex-Werte verglichen: Null Drift. Reine Namens-Substitution, kein Folge-Issue im DS-Repo noetig.

DS-Werte zwingen, App-Werte mit "behalten + Kommentar" stehen lassen — nicht noetig, weil drift-frei.

dashboard.css Tab-/Switcher-Block

Chosen (Research)

Vendor-CSS-Block (~50 LOC) ersatzlos entfernen. Sonst ueberdeckt die spaeter geladene Vendor-Regel gleicher Spezifitaet die DS-Defaults — Doppel-Klassen .tab-btn gat-tab waeren optisch wirkungslos.

Block behalten + Doppel-Klassen — sieht nach Phase 2 aus wie vorher, kein Konvergenz-Gewinn.

INK-Mapping in den Chart-Modulen

Chosen (Research)

ES-Modul-Bezug mit Alias INK as DS_INK, lokaler App-Adapter mappt PALETTE[*] auf semantische App-Rollen (green/blue/orange/red/soft/paper). Klare Trennung: INK.green = App-Rolle, DS_INK.text = DS-Tonklasse.

Naive Direkt-Uebernahme { INK } — App-Code greift auf INK.blue zu, das ist undefined, Diagramm-Serien werden grau.

HC-Toggle: Klasse auf <html> oder <body>?

Chosen (CONTEXT 3)

<head>-Inline-Skript setzt die Klasse auf documentElement (<body> existiert noch nicht). JS-Toggle in app.js spiegelt sie danach auch auf <body>. DS-Selector .gat-mode-hc body greift via Ancestor-Match auch bei <html class="gat-mode-hc">.

Nur <body> togglen — FOWT-Flash beim Reload mit aktivem HC.

VA_DECAL-Wert (Plan-Ist-Schraffur)

Akzeptiert

DS-Wert weicht von App-Wert ab (anderes dashArray, etwas weniger Alpha). CONTEXT-Deferred-3 sagt: "Uebernehmen, nicht diskutieren". Visualisierungs-Drift minimal, unter Snapshot-Schwelle.

Snapshots: Baseline + Final, nichts dazwischen

Chosen (CONTEXT 2)

Playwright/Chromium 1440x900, Fixture VA-2026-Auflage.pdf. Diff-Schwelle: <5 % in DS-bewussten Stellen, keine Layout-Brueche. Zwischen-Phasen-Snapshots sind Pflege ohne Mehrwert.

Verbleibende .web-*-Klassen -> .app-*

Chosen (CONTEXT 4)

Klare Namespace-Trennung. Konkret: nur .web-panel__fs-btn -> .app-panel-fs-btn (App-spezifischer Vollbild-Knopf, kein DS-Pendant). Funktionsklassen (.tab-btn, .switch-btn, .is-active, .tab-panel) bleiben unveraendert.
